1. Introduction.

Anything goes in. 
Anything goes out! 
Fish, bananas, old pyjamas, 
Mutton! Beef! and Trout!

[CHA 1989]

The programming language Orson [MOE 2018] can perform input and output on Unix streams [STE 2005]. A stream is an ordered series of zero or more unsigned bytes. Orson’s standard prelude binds the name stream to an execution type that describes a stream.
      Opening a stream associates it with a file. (A stream may also be associated with a peripheral device or a program, but that isn’t discussed here.) It may be opened as an input stream, which allows reading unsigned bytes from the file, or as an output stream, which allows writing unsigned bytes to the file. It may also be opened for both input and output. Closing an open stream breaks its association with a file. Reading or writing a closed stream has undefined results.
      Each unsigned byte in an open stream is identified by a nonnegative integer called its index. The stream’s position is the index of the next unsigned byte to be read or written. When a stream is read or written, its position is incremented. Its position may also be changed without reading or writing anything.
      Orson represents unsigned bytes as nonnegative integers. The standard prelude binds the name eos to a negative integer that may be read from an input stream. The integer eos does not represent an unsigned byte, but acts as a sentinel that represents the end of the stream.

2. The library file file.os.

The library file file.os provides forms that perform input and output on streams. Some of these forms read and write single characters. Others read a series of characters that notates an integer, a real, or a string. Still others write a series of characters that notates an integer, a real, a string, or a pointer. Orson uses a 31-bit character set that includes the familiar 21-bit Unicode [UNI 2006] and 7-bit ASCII [ANS 1986] character sets as subsets. Each character is read or written as a series of one to six unsigned bytes in the UTF-8 encoding [UNI 2006].
      The rest of this note discusses forms and other objects that are provided by file.os. They include the characters eol, eop, and ill, the streams errput, input and output, and the forms close, flush, get, index, open, opened, put, read, reset, unget, unread, write, and writeln. Many of these forms return Boolean values that tell whether they performed successfully. Most were suggested by procedures from the programming languages C [KER 1988] and Pascal [JEN 1974].
      Orson is currently implemented as a translator that converts Orson programs to GNU C programs [STA 2008] which run under Unix-like operating systems, such as GNU/Linux. As a result, most forms provided by file.os are actually just false beards for functions defined by the GNU C Library [LOO 2014]. They use the form c's [MOE 2017] to call these functions. Readers who wish more information may consult the Unix man pages for the functions that are called in this way. For example, information about the format strings used by write and writeln may be found in the man page for the function fprintf.
      GNU C uses objects called locales to specify conventions about input and output. All the forms defined by file.os use a default locale, called the C or POSIX locale. As a result, some forms (like those involving formatted input and output) may behave differently than expected. Their behavior may be changed by using c's to call the function setlocale, which changes the locale. Details can be found in the man page for setlocale.

2.1. eol, eop, and ill.

The names eol, eop, and ill are bound in the following way.

cha eol

 :− 

'\N'

cha eop

 :− 

'\F'

cha ill

 :− 

'\#FFFD'

The name eol is bound to an invisible character that terminates a line. Writing eol to a text file will end the current line and start a new one. The name eop is bound to an invisible character that some programs use to terminate a page. Writing eop to a text file will end the current page and start a new one. The name ill is bound to the Unicode replacement character '�'. It is used in place of a character whose value is unknown, or that cannot be represented in Unicode [UNI 2006].

2.2. errput, input, and output.

The names errput, input, and output are bound in the following way.

errput

 :− 

c's(stream, ''stderr'')

input

 :− 

c's(stream, ''stdin'')

output

 :− 

c's(stream, ''stdout'')

The names input and output are bound to streams that are associated, respectively, with a standard input device and a standard output device. The stream input is usually associated with the keyboard, and the stream output is usually associated with the console display. However, some programs may associate input and output with files.
      The name errput is bound to an unbuffered stream that is associated with the same standard output device as output. It is used to write error messages and other diagnostic information to the user.

2.3. close.

The name close is bound in the following way.

close :− 
c(form (stream s) bool: 
c(fc's(int, ''fclose'': s) = 0)

The form close takes a stream s as its single argument. If s is an output stream, then it is flushed. The form close then severs the association between s and a file or peripheral device.

2.4. flush.

The name flush is bound in the following way.

If flush is called with an open output stream s as its argument, and s has a buffer, then all bytes in the buffer will be written to s. The stream s remains open after it has been flushed in this way. If flush is called with no arguments, then it will flush all open output streams that belong to the current program.

2.5. get.

The name get is bound in the following way.

g((form (stream s) int:

If get is called with an open input stream s as its argument, then it will read the next unsigned byte from s and return it as an int. If no unsigned bytes remain to be read from s, then get will return eos instead. If get is called with no arguments, then it reads from the input stream input.

2.6. index.

The name index is bound in the following way.

index :−

i(fc's(int, ''ftell'': s))

If index is called with an open stream s as its argument, then it will return the current index of s.

The form open opens a file, and returns a stream that is associated with that file. It is usually called with two arguments, p and m. The string p is a pathname that denotes the file to be opened. The string m is a mode that specifies the input and operations which will be allowed on the file. Modes and their meanings are described in the following table.

 MODE 

 MEANING 

''a''

Open the file for writing. If the file does not exist, then create a new file. The stream is positioned at the end of the file.

''a+''

Open the file for reading and writing. If the file does not exist, then create a new file. The stream is positioned at the end of the file.

''r''

Open an existing file for reading. The stream is positioned at the beginning of the file.

''r+''

Open the file for both reading and writing. The stream is positioned at the beginning of the file.

''w''

Open the file for writing. If the file exists, then truncate it to zero length. If it does not exist, then create a new file. The stream is positioned at the beginning of the file.

''w+''

Open the file for reading and writing. If the file exists, then truncate it to zero length. If it does not exist, then create a new file. The stream is positioned at the beginning of the file.

If open is called with only p and m, then it will return a tuple with two slots, success and stream. If the Boolean slot success is true, then the file was opened successfully, and stream is the stream that is associated with the file. If success is false, then the file could not be opened with the requested mode, and stream is undefined.
      If open is called with three arguments s, p, and m, then it will return a Boolean value. If it returns true, then the file was opened successfully, and the variable s is set to the stream that is associated with the file. If it returns false, then the file could not be opened with the requested mode, and s is undefined.
      If p and m are missing, then they default to specify a temporary file that allows reading and writing. The temporary file will be deleted automatically when its stream is closed.

The form opened is a wrapper that acts like a call to open followed by a call to close. The for clause (for bool b, stream s in opened(p, m) do d) makes a stream that is associated with a file whose pathname is the string p. The stream allows input and output operations described by the string m, its mode.
      If the file was opened successfully, then d is executed with b bound to true and s bound to the stream. After d has finished executing, the stream is closed. The for clause returns true if the stream was closed successfully, and returns false otherwise. If d terminates the for clause by calling throw, then the stream will not be closed.
      If the file was not opened successfully, then d is executed with b bound to false and s bound to an undefined stream. After d has finished executing, the stream is not closed, and the for clause simply returns false.
      If p and m are missing, then they default to specify a temporary file that allows reading and writing. The temporary file will be deleted automatically when it is closed.

2.9. put.

The name put is bound in the following way.

put :− 
p(alt

If put is called with a output stream s and an integer b that represents an unsigned byte, then it will write b to s at its current index. If the stream argument s does not appear, then the stream output is assumed to appear in its place.

If read is called with an input stream s as its only argument, or if it is called with a stream s and the type char, then it will read a Unicode [UNI 2006] character from s and return it as an int. (This is done by a secret procedure Read, whose definition is not shown.) If there are no characters that remain to be read from s, then read will return eos. If it reads a character with an illegal UTF-8 encoding [UNI 2006], then it will return ill.
      If read is called with a stream s, a format string f, and a list of variables l, then it will read the notations of objects specified by f from s, and assign the variables in l to those objects. The number of bytes read from s is then returned. Results are undefined if read encounters a series of characters that do not notate the objects specified by f.
      In all the above cases, if the stream argument s does not appear, then the stream input is assumed to appear in its place.

2.11. reset.

The name reset is bound in the following way.

reset :− 
r(alt 
r((form (stream s) bool: 
r((fc's(int, ''fseek'': s, 0, c's(int, ''SEEK_END'')) = 0), 
r((form (stream s, int i) bool: 
r((fc's(int, ''fseek'': s, i, c's(int, ''SEEK_SET'')) = 0))

If reset is called with a stream s and a nonnegative integer i as its arguments, then it will reset the index of s to i. For example, reset(s, 0) will reposition s to its first byte.
      If the argument i does not appear, then the maximum index of s is assumed to appear in its place. As a result, reset(s) positions the stream s to its end. This is somewhat nonintuitive, and is the opposite of what the procedure reset did in Pascal [JEN 1974].

2.12. unget.

The name unget is bound in the following way.

If unget is called with an input stream s and an integer b that represents an unsigned byte, then it sends b back to s so that it may be read again. Calling unget two or more times in a row on the same input stream has undefined results. If the stream argument s does not appear, then the stream input is assumed to appear in its place.

3. Extension conventions.

Some Orson programs define representations for objects and methods that operate on those objects. For example, a program might define a type comp to represent complex numbers, and define methods like "+", "−", "×", and "/" to operate on them. The program might then be placed in a library so it could be used as part of larger programs.
      By convention, a program that defines a type t and its operations may also extend the form read so it can read notations for objects of type t. For example, the complex number program may define read(s, comp) to read a notation like 2.0i + 3.0 from a stream s, then return the corresponding complex number. Similarly, the program may extend the form write so it can write notations for objects of type t. For example, the complex number program may define write(s, z) so it works when z is a comp.
      Also by convention, if read(s, t) reads a notation from s, then read(t) may read the same notation from input. Similarly, if write(s, o) writes a notation to s, then write(o) may write the same notation to output. The form writeln may be extended so that it behaves the same way as write, except that it terminates its output with eol.
